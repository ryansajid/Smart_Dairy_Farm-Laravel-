## 1. Introduction

### 1.1 Purpose

This document establishes the standardized format and procedures for documenting meeting minutes and tracking action items for the Smart Dairy Ltd. Smart Web Portal System project. Effective meeting documentation ensures:

- **Accountability**: Clear record of decisions made and commitments undertaken
- **Traceability**: Historical record of project discussions and decisions
- **Communication**: Consistent information sharing across the project team
- **Progress Tracking**: Monitoring of action item completion
- **Knowledge Retention**: Preserved institutional knowledge for future reference

### 1.2 Scope

This template applies to all project meetings including but not limited to:
- Internal team meetings
- Stakeholder meetings
- Technical review sessions
- Vendor discussions
- Status review meetings

### 1.3 Definitions

| Term | Definition |
|------|------------|
| **Meeting Minutes** | Formal written record of meeting proceedings |
| **Action Item** | Specific task assigned to an individual with a due date |
| **Attendee** | Person present at the meeting |
| **Absent** | Expected participant who did not attend |
| **Decision** | Resolution reached during the meeting |
| **Follow-up** | Additional action required after the meeting |

## 2. Meeting Types

### 2.1 Daily Standups

| Aspect | Description |
|--------|-------------|
| **Frequency** | Daily |
| **Duration** | 15 minutes maximum |
| **Purpose** | Share progress, identify blockers, coordinate daily work |
| **Required Attendees** | Development team, Scrum Master |
| **Optional Attendees** | Project Manager, Product Owner |

**Key Elements:**
- What was accomplished yesterday?
- What is planned for today?
- Any blockers or impediments?

### 2.2 Sprint Planning

| Aspect | Description |
|--------|-------------|
| **Frequency** | At the beginning of each sprint (bi-weekly) |
| **Duration** | 2-4 hours |
| **Purpose** | Define sprint goals and commit to deliverables |
| **Required Attendees** | Development team, Scrum Master, Product Owner |
| **Optional Attendees** | Project Manager |

**Key Elements:**
- Review of product backlog
- Sprint goal definition
- Task breakdown and estimation
- Team capacity planning
- Sprint commitment

### 2.3 Sprint Review

| Aspect | Description |
|--------|-------------|
| **Frequency** | End of each sprint (bi-weekly) |
| **Duration** | 1-2 hours |
| **Purpose** | Demonstrate completed work, gather feedback |
| **Required Attendees** | Development team, Scrum Master, Product Owner |
| **Optional Attendees** | Stakeholders, Project Manager |

**Key Elements:**
- Sprint goal achievement review
- Product increment demonstration
- Stakeholder feedback collection
- Product backlog refinement

### 2.4 Retrospectives

| Aspect | Description |
|--------|-------------|
| **Frequency** | End of each sprint (bi-weekly) |
| **Duration** | 1-2 hours |
| **Purpose** | Continuous improvement, team reflection |
| **Required Attendees** | Development team, Scrum Master |
| **Optional Attendees** | Product Owner |

**Key Elements:**
- What went well?
- What could be improved?
- Action items for improvement
- Team morale check

### 2.5 Stakeholder Meetings

| Aspect | Description |
|--------|-------------|
| **Frequency** | Monthly or as needed |
| **Duration** | 1-2 hours |
| **Purpose** | Status reporting, strategic alignment, issue escalation |
| **Required Attendees** | Project Manager, Key Stakeholders |
| **Optional Attendees** | Technical Lead, Product Owner |

**Key Elements:**
- Executive summary
- Budget and schedule status
- Risk and issue overview
- Strategic decisions
- Resource requirements

### 2.6 Technical Reviews

| Aspect | Description |
|--------|-------------|
| **Frequency** | As needed, typically bi-weekly |
| **Duration** | 1-2 hours |
| **Purpose** | Architecture decisions, code reviews, technical planning |
| **Required Attendees** | Technical Lead, Development Team, Solution Architect |
| **Optional Attendees** | Project Manager, QA Lead |

**Key Elements:**
- Architecture discussions
- Technology evaluations
- Technical risk assessment
- Code review outcomes
- Technical debt review

### 4.2 Action Item Attributes

| Attribute | Description | Required |
|-----------|-------------|----------|
| **ID** | Unique identifier (e.g., AI-001) | Yes |
| **Description** | Clear, concise task description | Yes |
| **Owner** | Single person responsible | Yes |
| **Due Date** | Target completion date | Yes |
| **Status** | Current status | Yes |
| **Priority** | High / Medium / Low | Yes |
| **Source** | Meeting ID where created | Yes |
| **Created Date** | Date action item assigned | Yes |

| **Notes** | Additional information | Optional |
| **Dependencies** | Other action items dependent on | Optional |

### 4.3 Follow-up Procedures

#### 4.3.1 Daily Monitoring
- Scrum Master reviews open action items daily
- Blocked items escalated immediately
- Owner updates status in tracking system

#### 4.3.2 Weekly Review
- Project Manager reviews all action items weekly
- Overdue items flagged for discussion
- Resource conflicts identified and resolved

#### 4.3.3 Escalation Path

| Level | Condition | Action |
|-------|-----------|--------|
| 1 | Due within 2 days, no progress | Reminder notification |
| 2 | Due date passed | Direct follow-up with owner |
| 3 | Overdue > 3 days | Escalate to Project Manager |
| 4 | Overdue > 1 week | Escalate to Project Sponsor |

#### 4.3.4 Closure Criteria
An action item may be closed when:
- Work is completed and deliverable verified
- Owner confirms completion in tracking system
- Project Manager or designee approves closure
- For cancelled items, reason documented

## 9. Best Practices

### 9.1 Effective Meeting Minutes Guidelines

#### Before the Meeting
- [ ] Review previous minutes for context
- [ ] Obtain agenda in advance
- [ ] Prepare minutes template
- [ ] Set up recording (if permitted)
- [ ] Confirm attendee list

#### During the Meeting
- [ ] Record basic information (date, time, attendees)
- [ ] Follow agenda structure
- [ ] Capture key decisions and rationale
- [ ] Note action items with owners and dates
- [ ] Ask for clarification when needed
- [ ] Mark items requiring follow-up

#### After the Meeting
- [ ] Review and clean up notes within 2 hours
- [ ] Distribute draft to key participants for review
- [ ] Finalize and distribute within 24 hours
- [ ] Enter action items into tracking system
- [ ] Archive in appropriate repository

### 9.2 Action Item Best Practices

#### Writing Effective Action Items

âœ… **Good Examples:**
- "Create database schema document for User Management module by Feb 5"
- "Schedule vendor meeting with Cloud provider for pricing discussion by Feb 3"
- "Update API documentation with new authentication endpoints by Feb 10"

âŒ **Poor Examples:**
- "Work on database stuff"
- "Call the vendor"
- "Fix documentation"

#### Characteristics of Good Action Items

| Characteristic | Description |
|----------------|-------------|
| **Specific** | Clear description of what needs to be done |
| **Measurable** | Criteria for completion defined |
| **Achievable** | Realistic given resources and constraints |
| **Relevant** | Contributes to project objectives |
| **Time-bound** | Specific due date assigned |

### 9.3 Common Pitfalls to Avoid

| Pitfall | Impact | Mitigation |
|---------|--------|------------|
| Vague action items | Unclear expectations | Use SMART criteria |
| Multiple owners | Accountability gaps | Single owner per item |
| Missing due dates | Priority confusion | Always include deadlines |
